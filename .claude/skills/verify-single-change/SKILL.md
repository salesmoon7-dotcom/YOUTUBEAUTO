# 한 번에 하나만 변경 규칙 검증

## Purpose

CLAUDE.md §6.2 "한 번에 여러 곳을 동시에 수정 (원인 파악 불가능해짐)" 규칙을 검증합니다:

1. **변경 범위 확인** — 한 세션에서 수정된 파일이 하나의 목적에 집중되었는지
2. **관련 없는 변경 탐지** — 버그 수정 중 무관한 리팩토링이 포함되지 않았는지
3. **변경 크기 확인** — 한 번의 수정이 과도하게 크지 않은지

## When to Run

- 코드 수정 후 커밋 전
- 여러 파일을 수정한 후
- 버그 수정 시 (리팩토링 혼재 방지)

## Workflow

### Step 1: 변경된 파일 수집

```bash
git diff HEAD --name-only 2>/dev/null
git diff --cached --name-only 2>/dev/null
```

### Step 2: 변경 파일 수 확인

**기준:**
- 1-3개 파일: 정상 범위
- 4-6개 파일: 주의 필요 — 모두 같은 기능과 관련된지 확인
- 7개 이상: 경고 — 여러 기능을 동시에 수정한 가능성 높음

### Step 3: 변경 목적 일관성 확인

변경된 파일들이 하나의 목적을 공유하는지 분석합니다:

```bash
# 각 파일의 변경 내용 요약
git diff HEAD -- <file> | head -30
```

**일관성 판단 기준:**
- 같은 함수를 여러 파일에서 호출하는 변경 → 일관적 (OK)
- 관련 없는 함수의 리팩토링이 포함된 변경 → 비일관적 (FAIL)
- 버그 수정 + 코드 정리가 혼재 → 비일관적 (FAIL)

### Step 4: 버그 수정 시 리팩토링 혼재 탐지

변경 내용에서 다음 패턴을 확인합니다:

- 변수 이름 변경 (rename) — 버그 수정과 무관한 경우 FAIL
- import 정리 — 버그 수정과 무관한 경우 WARN
- 포매팅 변경 — 기능 변경과 분리해야 함

**PASS 기준:** 모든 변경이 하나의 목적에 집중됨
**FAIL 기준:** 관련 없는 변경이 혼재됨

### Step 5: 결과 보고

```markdown
## verify-single-change 결과

| 검사 | 상태 | 상세 |
|------|------|------|
| 변경 파일 수 | OK/WARN | N개 |
| 목적 일관성 | PASS/FAIL | 설명 |
| 리팩토링 혼재 | PASS/FAIL | 설명 |
```

## Output Format

| 항목 | 값 | 상태 |
|------|-----|------|
| 변경 파일 수 | N | OK/WARN/FAIL |
| 목적 | 설명 | 일관/비일관 |
| 혼재 여부 | 설명 | PASS/FAIL |

## Exceptions

다음은 **위반이 아닙니다**:

1. **하나의 기능이 여러 파일에 걸친 경우** — 예: pipeline.py + sub_runners.py + scripts/seaart_automation.py를 같이 수정하는 것은 이미지 파이프라인이라는 하나의 목적
2. **호출자-피호출자 함께 수정** — 함수 시그니처 변경 시 호출하는 코드도 함께 수정해야 함
3. **verify 스킬 생성/업데이트** — .claude/skills/ 파일과 CLAUDE.md를 함께 수정하는 것은 스킬 관리라는 하나의 목적
4. **일괄 @DEBUG_PIPELINE 제거** — 안정화 완료 후 여러 파일에서 디버그 코드를 제거하는 것은 정상
